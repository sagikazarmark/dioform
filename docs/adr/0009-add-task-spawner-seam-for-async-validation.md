# Add a task-spawner seam for async validation

Dioform will route adapter-side **Async Validation** task spawning and cancellation through a small `TaskSpawner` seam (`spawn`, `cancel`) held as `Rc<dyn TaskSpawner>` on the **Form Handle**, with a Dioxus adapter for production and an inline adapter for tests. This supersedes the previous in-code decision (recorded as a comment in `adapter_runtime.rs`) that the runtime boundary would be "the core begin/complete/schedule API plus this adapter execution layer, not a pluggable runtime trait."

The seam is deliberately narrow. Timers already vary across a real seam: the delay future is injected by the application through a delay-factory closure (`debounce_duration`), and tests already substitute manual delays, so timers stay outside `TaskSpawner`. The only hard-wired coupling is task spawning: `dioxus_core::spawn`, `dioxus_core::remove_future`, and `Runtime::current().current_task()`. Threading an explicit spawner-owned task id through `spawn` removes the `Runtime::current()` identity lookup, so the seam simplifies the completion path rather than only abstracting it.

Two adapters justify the seam: the Dioxus spawner in production and an inline spawner in tests. The inline spawner lets adapter async-validation logic (start, cancel, stale-result protection) be exercised on a bare **Form Handle** and lets tests assert real cancellation instead of inferring it by drop-counting perpetually-pending fake futures. Hook-level tests still require a Dioxus component scope; the seam does not change that. `TaskSpawner` is carried as `dyn` rather than a type parameter to avoid widening the **Form Handle** generics across every binding and hook.
