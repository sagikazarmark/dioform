# Carve Form Core into a field store, submission state, and chain executor

The **Form Core** state machine currently lives as one struct with ~229 methods that hold field state in parallel `BTreeMap`s keyed by **Field Identity** (`field_versions`, `field_metadata`, `collection_states`) plus loose submission fields, with consistency maintained by convention across every method. Dioform will carve three internal modules out of it, co-designed because their state interlocks. This does not change public API, crate boundaries, or the renderer-agnostic guarantee; it is an internal restructuring of `dioform-core`.

A **Field Store** becomes the single owner of field-keyed state (version, metadata, and collection identity) keyed by **Field Identity**. It preserves lazy **Field Registration**: reads do not allocate and an unregistered field reads as version `0`, so the `unwrap_or_default()` convention that today appears at every read site collapses into one method. Version has exactly one owner, which is the property the other two modules depend on. Per-validator state (keyed by `(FieldIdentity, ValidatorId)`) stays in the existing `ValidationChainRegistry`; the **Field Store** stays field-keyed only.

A **Submission State** owns the submit-attempt, in-flight, last-status, and submit-validation-intent fields plus `submit_errors`. The scattered **Submit Availability** predicates split by ownership rather than moving wholesale into the **Submission State**: the chain-iterating queries move onto `ValidationChainRegistry`, where the validators already live, the **Submission State** owns the submit-error checks over its own stored errors, and **Form Core** composes `Submit Availability` from both. This keeps each module reading only the state it holds. **Submit Intent** stays stored erased inside the **Submission State**; the typed intent is applied only at the intent-scoped boundary, preserving ADR-0004.

A **Chain Executor** owns the sync-before-async sequencing and async planning that currently sit inline in **Form Core**, returning a side-effect-free `ChainOutcome` (sync transitions applied, async keys to start, async keys to skip). Spawning stays adapter-side behind the `TaskSpawner` seam (ADR-0009), and `Form Observer` events are mapped from the outcome by the caller, so the executor produces results without side effects. This carve is the most speculative of the three: `SourceState` and `ValidationChainRegistry` are already well-factored, so the executor only relocates ordering out of **Form Core** and should land after the **Field Store** gives version staleness a single owner. Every carved module is a reader of `FieldStore::version`, never a writer.
